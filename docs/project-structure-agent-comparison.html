Merchant Automation 架构雷达

这个项目的价值不在于“让 Agent 点网页”本身，而在于把自然语言、账号门店、执行 Recipe、安全门控、轨迹沉淀和 Dashboard 治理拆成了清晰边界。Agent 负责探索，Recipe 负责复用，Preflight 负责刹车。

核心架构 Operation + Recipe 浏览器层 browser-use 服务层 FastAPI + 飞书 数据层 SQLite stores

Highlights

四个最值得保留的亮点

这些亮点让项目从“能跑一次的自动化脚本”变成“可以运营、审计、逐步放权的自动化系统”。

A

契约先行

OperationContract 定义参数、成功标准和禁止行为。Agent 不再凭感觉执行，而是在业务契约里行动。

B

分层执行

ExecutionRouter 把确定性 Recipe、Agent 兜底、无执行器失败分开。热路径快，冷路径能探索。

C

安全门控

Preflight 将 commit 变成受全局开关、账号门店权限、Recipe 状态和成功率约束的显式决策。

D

闭环沉淀

synthesizer.py 能把 browser-use 历史转成 Recipe 步骤，成功探索可以回流为后续确定性执行资产。

Recipe flywheel

为什么这个结构比直接让 Agent 操作更稳

Agent 的强项是探索未知页面；系统的强项是把成功经验变成可审、可控、可复用的执行路径。两者结合后，成本和不确定性都会向热路径收敛。

Agent 探索
Recipe 回放
自然语言任务飞书或 CLI 进入规划服务
无步骤时探索AgentExplorer 驱动浏览器
成功后合成AgentHistoryList 到 RecipeDefinition
人工审核晋级candidate 到 prepare_ready / commit_ready
热路径回放RecipeStepExecutor 确定性执行
1

冷启动保留 Agent 弹性

页面没见过、Recipe 不存在或步骤刚失效时，Agent 可以重新探索后台，不需要人工先手写完整脚本。

2

成功历史变成结构化资产

合成器只抽取 navigate、click、input、upload_file 等关键动作，并用语义 target 和参数模板替代临时 DOM index。

3

审核与门控控制生产风险

candidate 可以先在 prepare 中验证；只有状态、权限、成功率都满足时才允许真实 commit。

4

失败能解释，也能回退

Trace、FailureAnalyzer、Dashboard 和最近失败窗口让系统知道为什么失败，而不是只返回“Agent 没完成”。

Skills vs Playwright

Codex / Claude Code 用 Skills，和调用 Playwright 的区别

它们不是替代关系。Skills 更像 Agent 的“工作方法和领域手册”，Playwright 更像浏览器的“确定性执行器”。当前项目里的 Recipe 层，正好处在两者之间。

维度 Codex / Claude Code Skills 直接调用 Playwright 当前项目的 Recipe / browser-use
核心定位 方法层

把重复的工作流、检查清单、领域知识、脚本和参考资料包装给 Agent，帮助它更可靠地思考和执行。

 工具层

提供浏览器自动化 API，适合确定性点击、填写、断言、截图和端到端测试。

 业务执行层

用 Operation 契约和 Recipe 步骤表达商家后台操作，既能由 Agent 生成，也能被系统治理。
适合放什么

项目启动步骤、测试策略、PR 审核流程、Recipe 设计规范、飞书卡片文案规范、故障排查路径。

稳定页面的浏览器动作、可重复验收、视觉回归、表单流程、登录态验证、Dashboard UI 回归。

平台、门店、操作类型、参数模板、提交前停止点、Recipe 状态、成功率与失败轨迹。
最大优势

跨任务复用经验，让不同 Agent 更像同一个团队成员；长文档按需加载，不必每次塞进上下文。

执行结果可重复、可测试、可并入 CI；页面行为清晰时，比自由探索便宜且稳定。

把探索成功转为生产资产，并用 candidate、prepare_ready、commit_ready 逐级放权。
主要边界

Skills 本身不等于浏览器控制能力。它能告诉 Agent 怎么做，但仍需要 shell、MCP、browser-use、Playwright 等工具落地。

Playwright 不理解业务意图。页面结构变动、语义含糊、跨平台后台差异，需要上层 Agent 或 Recipe 适配。

Recipe 合成质量取决于 Agent 历史质量；语义 target 跨后台变化仍需降级探索和人工审核兜底。
在本项目中的建议

沉淀为 repo 级 Skills：如何新增 Operation、如何审核 Recipe、如何处理真实后台失败、如何跑灰度。

用于 Dashboard 回归、Recipe 本地测试页面、登录流程验收和关键路径截图，不建议直接取代 Recipe 治理层。

继续让生产后台操作走 Operation + Recipe + Preflight；Agent 和 Playwright 都作为生成、验证、调试的辅助能力。

Positioning

三类能力的最佳落位

把能力放到正确层级，项目就不会退化成一堆浏览器脚本，也不会把生产执行完全交给不可解释的自由 Agent。

Skills：让 Agent 会做事

  • 写在 .agents/skills 或对应工具的技能目录里，适合团队规范和长流程。
  • 把“新增一个商家后台操作”做成固定流程：定义契约、补 Recipe、加测试、验 Dashboard。
  • 把线上故障处理做成技能：先读 trace，再判定 preflight、Recipe 过期、选择器失效或账号问题。

Playwright：让浏览器可验证

  • 适合稳定、可断言、可截图的场景，比如 Dashboard 和本地测试后台。
  • 可作为 CI 里的 UI 回归工具，验证页面是否渲染、按钮是否可点、流程是否走通。
  • 用于辅助验证 Recipe，而不是把业务状态机藏进脚本里。

Recipe：让自动化可运营

  • Recipe 是项目的生产资产，承载参数模板、提交前停止点和晋级状态。
  • Agent 成功探索后自动合成 candidate，后续由 Dashboard 审核和 Preflight 放权。
  • 失败时回退 Agent 探索，并把新经验沉淀回来，形成可观测飞轮。

Recommended architecture

推荐组合：Skills 管方法，Recipe 管业务，Playwright 管验证

这套项目结构已经接近这个方向。下一步最有价值的不是把所有东西改成 Playwright，而是把可复用经验沉淀到 Skill 和 Recipe 两个层级：前者服务开发 Agent，后者服务生产执行。

Agent Skills
编码流程、故障诊断、Recipe 审核、测试与发布规范
Operation
业务意图、参数 schema、成功标准、禁止行为
Recipe
可回放步骤、状态机、成功率、人工晋级
browser-use
未知页面探索、语义定位、动作历史输出
Playwright
本地页面验收、Dashboard 回归、截图和浏览器断言
